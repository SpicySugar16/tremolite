# 任务分解 (Task Decomposition)

将复杂任务拆解为可执行的小步骤，建立依赖关系，估算工作量。

## 分解方法

1. **目标定义** — 任务完成后「看起来是什么样」，最小可验收标准
2. **逆向分解** — 从最终输出反推需要的环节（输出 → 处理 → 输入）
3. **粒度控制** — 每个子任务不超过 1 天工作量，超则继续拆分
4. **依赖编排** — 明确前置条件和阻塞关系（DAG 而非线性列表）
5. **风险标注** — 标注不确定性高的子任务（技术探索、未接触过的 API、第三方库集成）

## 输出格式

```
任务: [名称]
  优先级: P0/P1/P2/P3
  估算: 3-5天
  子步骤:
    [✓] 步骤 1 — 准备工作（已做）
    [ ] 步骤 2 — 核心实现（2天，依赖步骤1）
        - 子项 A（已完成）
        - 子项 B（进行中）
    [ ] 步骤 3 — 测试（1天，依赖步骤2）
```

## 原则

- 技术探索类任务单独列一阶段，不做完不放进度
- 每一步完成后主动汇报——跳过汇报让人感到失控
- 发现问题不解释不道歉，直接做行动计划
