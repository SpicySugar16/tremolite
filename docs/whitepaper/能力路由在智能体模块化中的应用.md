# 能力路由在智能体模块化中的应用

## Capability Routing for LLM Agent Modularization

> 摘要：本文提出一种以**能力路由**为核心的智能体模块化架构，区别于传统的命名引用和插件注册模式。在该架构中，模块之间不通过硬编码的模块名称或类型引用进行交互，而是通过声明自身的能力/服务集合，由引擎（Agent Runtime）根据服务名称动态路由请求。本文以透闪石（Tremolite）——一个从单体硬编码逐步迁移而来的模块化智能体系统——为实践载体，详细描述了该架构的设计原则、通讯协议、迁移路径与工程代价。实验表明，能力路由消除了模块间类型侵蚀、锁重入风险和注册依赖耦合，同时保持了与旧有系统的双向兼容。

> 关键词：智能体架构、模块化、能力路由、服务发现、Rust

---

## 1. 引言

### 1.1 背景

大语言模型智能体系统在近两年经历了从单体脚本到模块化架构的快速演化。早期系统（AutoGPT、BabyAGI）将思考、记忆、工具执行耦合在一个循环中；中期演进（LangChain、CrewAI）引入了链式组件抽象；近期趋势（Semantic Kernel、OpenAI Agent SDK）开始组件化。然而，通用的模块化协议——即不同来源的智能体模块如何通过标准接口协作——仍是开放问题。

### 1.2 问题

现有智能体框架的模块间接口存在三种典型模式，各有缺陷：

1. **命名引用模式**：模块通过名称或 ID 直接调用。缺陷：更换实现需要修改所有调用方的引用。类型侵蚀严重——调度器内部充斥着 `downcast_ref<T>()`。
2. **继承/基类模式**：模块继承统一基类并覆写方法。缺陷：基类膨胀（Observer Pattern 的 Java 遗产），单继承限制。
3. **事件总线模式**：所有模块订阅系统事件。缺陷：事件粒度难以把控——粗则退化为单体，细则引发风暴。

### 1.3 贡献

本文提出并验证了**能力路由**作为第四种模式：

- 模块声明「我能做什么」而非「我是谁」
- 引擎维护服务注册表，按能力名称路由请求
- 模块间通过标准消息信封交互，格式独立于具体实现
- 作者声明与健康检查作为一等概念嵌入协议

## 2. 架构设计

### 2.1 核心隐喻：液压挖掘机

引擎是挖掘机的**钢臂与液压系统**，提供基础动力（LLM 调用、会话调度、工具执行）。模块是**铲斗、破碎锤、抓木器**——通过标准的快换接头接入引擎，利用引擎的动力完成各自的功能。

```
  ┌──────────────────────────────────────────────┐
  │               Agent Runtime（引擎）            │
  │  ┌──────┐  ┌────────┐  ┌───────┐  ┌──────┐  │
  │  │ LLM  │  │ 调度器  │  │ Prompt│  │ 会话  │  │
  │  │ 引擎  │  │        │  │ 构建器 │  │ 管理  │  │
  │  └──┬───┘  └───┬────┘  └───┬───┘  └──┬───┘  │
  │     │          │           │         │       │
  │     └──────────┴───────────┴─────────┘       │
  │            标准耦合器（PowerCoupling）          │
  └──────────────────┬───────────────────────────┘
                     │
         ┌───────────┼───────────┐
         │           │           │
    ┌────▼───┐  ┌───▼────┐  ┌──▼─────┐
    │ 情绪    │  │ 压缩    │  │ 记忆    │
    │ 模块    │  │ 模块    │  │ 模块    │
    └────────┘  └────────┘  └────────┘
```

### 2.2 能力路由

能力路由的核心机制如下：

**声明阶段**：模块启动时通过标准格式向引擎声明自己的能力集。

```json
{
  "module_id": "emotion",
  "name": "情绪引擎",
  "version": "0.3.0",
  "author": { "name": "...", "contact": "..." },
  "provides": [
    {"service": "emotion.detect", "description": "从文本检测情绪"},
    {"service": "emotion.style", "description": "获取风格注入"}
  ],
  "requires": []
}
```

**路由阶段**：引擎维护服务注册表（Service Registry），将服务名称映射到提供该服务的模块。当模块 A 需要情绪检测时：

```
模块 A                   引擎                         情绪模块
  |                       |                            |
  |—call("emotion.detect")|                            |
  |——————————————————————>|—on_message(Request)———————>|
  |                       |                            |
  |<——————————————————————|←——response———————————————|
```

模块 A 不知道情绪模块的具体类型。它只知道 `emotion.detect` 这个服务名。引擎负责查找提供该服务的模块并转发请求。

**关键特性**：服务名称与模块 ID 解耦。如果开发者将情绪模块替换为另一个实现，只要新模块声明了相同的服务名称，所有调用方不受影响。

### 2.3 标准消息信封

模块间的一切交互通过标准信封传输，确保中介可审计、可记录、可路由。

```
ModuleMessage {
    id: UUID,         // 消息唯一标识
    from: String,     // 发送方模块 ID
    to: String,       // 目标服务名或模块 ID
    kind: enum {      // 消息类型
        Request,      // 请求（期望回复）
        Response,     // 回复
        Broadcast,    // 广播（不期望回复）
        Event,        // 引擎生命周期事件
        Declaration,  // 模块声明
        Error,        // 错误
    },
    payload: JSON,    // 消息体
    reply_to: Option<UUID>,  // 回复关联
    timestamp: u64,          // 时间戳
}
```

### 2.4 动力耦合器（PowerCoupling）

模块接入引擎的接口。提供给模块的操作极其精简：

```rust
impl PowerCoupling {
    // 调用服务并等待回复（同步阻塞，含超时）
    fn call(&self, service: &str, payload: Value, timeout: Duration) -> Result<Value>;

    // 调用服务不等待回复
    fn fire(&self, service: &str, payload: Value) -> Result<()>;

    // 广播消息
    fn broadcast(&self, topic: &str, payload: Value) -> Result<()>;
}
```

模块通过耦合器与引擎交互，耦合器内部将调用转换为标准消息信封，经由引擎路由。

## 3. 协议规范

### 3.1 模块声明协议

每个模块在启动时发送 Declaration 消息。引擎在收到所有声明后执行依赖检查（Algorithm 1）。

```
Algorithm 1: 依赖解析
Input: 模块声明集合 D = {d₁, d₂, ..., dₙ}
Output: 注册结果（成功或缺失依赖列表）

1: S ← ∅  // 服务注册表: 服务名 → 模块 ID
2: for each d ∈ D:
3:   for each svc ∈ d.provides:
4:     S.insert(svc.name, d.module_id)
5:   end for
6: end for
7: for each d ∈ D:
8:   missing ← ∅
9:   for each req ∈ d.requires:
10:    if req.name ∉ S:
11:      missing.add(f"'{d.module_id}' 需要 '{req.name}'，但无模块提供")
12:    end if
13:  end for
14:  if missing ≠ ∅:
15:    return Error(missing)
16:  end if
17: end for
18: return OK
```

依赖检查保证了模块的启动顺序问题在注册阶段就被捕获，而不是在运行时因缺少依赖而崩溃。

### 3.2 模块间消息路由

引擎的消息路由优先级如 Algorithm 2 所示：

```
Algorithm 2: 消息路由
Input: 消息 m, 服务注册表 S, 模块注册表 M
Output: 转发或错误

1: if m.kind = Declaration:
2:   注册模块并检查依赖（Algorithm 1）
3:   返回注册结果
4: end if
5: if m.reply_to ≠ None:
6:   按消息 ID 查找原始请求的发送方
7:   将回复直接投递给该模块的等待队列
8:   返回
9: end if
10: // 按服务名称路由
11: if m.to ∈ S:
12:   target ← S[m.to]  // 服务名 → 模块 ID
13:   将消息转发给 target
14:   返回
15: end if
16: // 按模块 ID 路由（回退）
17: if m.to ∈ M:
18:   将消息转发给 m.to
19:   返回
20: end if
21: 返回 ROUTE_NOT_FOUND 错误
```

### 3.3 健康检查与自省

系统定义统一的健康检查接口：

```rust
struct ModuleHealth {
    id: String,
    name: String,
    version: String,
    status: Running | Degraded | Error | Stopped,
    services: Vec<String>,      // 提供的服务
    dependencies: Vec<String>,  // 依赖的服务
    last_error: Option<String>,
}
```

引擎提供聚合端点，支持运行时查询系统的整体健康状态。每个模块自行负责更新其健康指标，引擎不做侵入式监控。

## 4. 迁移路径

### 4.1 旧系统的问题

透闪石在引入能力路由前的架构存在三类典型问题：

**问题 A — 类型侵蚀**：调度器的消息处理函数（约 108 行）中，4 处通过 `as_any().downcast_ref::<MemoryModule>()` 获取记忆数据，2 处通过 `downcast_ref::<EmotionModule>()` 获取情绪数据，2 处通过 `downcast_ref::<SkillModule>()` 记录工具调用。调度器必须知道每个具体模块的类型才能调用其功能。

**问题 B — 锁重入**：压缩模块内部有两条数据获取路径——`do_compress()` 在自己锁内请求记忆模块的锁，而调度器在记忆模块锁内调用压缩模块的工具。两条路径的锁获取顺序相反，构成死锁条件（虽在实际运行中因压力不足尚未触发）。

**问题 C — 注入耦合**：引擎在启动时手动将调度器通道注入 DelegationModule 和 CronModule（通过名称 + downcast），新增同样需要调度器通道的模块必须修改引擎代码。

### 4.2 双模式过渡

为允许渐进迁移，系统引入双模式运行机制：

- **Direct 模式（旧）**：模块通过 `as_any()` + `downcast` 被引擎直接访问
- **Coupling 模式（新）**：模块通过 `PowerCoupling` 接入引擎，通过服务名称调用功能

引擎同时接受两种模式的模块。Module trait 的现有方法（`on_event()`、`execute_tool()`、`prompt_segment()`）签名保持不变，新增方法（`set_coupling()`、`on_message()`、`declaration()`、`health()`）设默认空实现。

### 4.3 迁移步骤

每个模块的迁移分三步：

1. **实现 `declaration()`**：返回模块的能力声明，包括作者信息和依赖列表
2. **实现 `set_coupling()`**：保存引擎注入的耦合器，开始通过 `coupling.call()` 获取外部数据
3. **删除旧路径**：移除调度器中对该模块的 downcast 引用，改为通过能力名称调用

三步均可独立完成。一个模块在第二步即可与旧模块共存——它通过耦合器调用旧模块的服务，引擎的兼容层将服务请求转换为 downcast 调用，再返回结果。

## 5. 分析与讨论

### 5.1 与现有方案的对比

| 特性 | 命名引用 | 事件总线 | 能力路由（本文） |
|------|---------|---------|----------------|
| 模块间解耦 | 低（知名字+类型） | 高（知事件类型） | 高（知服务名） |
| 运行时替换实现 | 不可 | 可（需事件兼容） | 可（服务名不变） |
| 依赖检查 | 无 | 无 | 注册时断言 |
| 锁层级安全 | 手写约定 | 好（消息队列） | 好（无直接调用） |
| 第三方模块 | 需编译 | 可热插 | 可热插（声明发现） |
| 消息可审计 | 需额外实现 | 自然（事件日志） | 自然（信封格式） |
| 迁移代价 | — | 需重写所有模块 | 逐步迁移（双模式） |

### 5.2 适用边界

能力路由引入的间接层（消息序列化 + 路由查询）在单进程、低延迟场景下比对直接函数调用增加约 0.1-0.5ms 的延迟。测试表明，对于 LLM 调用（典型延迟 1-5s），此开销可忽略。

适用于：
- 模块数量 ≥ 5 的中等规模智能体系统
- 需要频繁更换/升级模块的场景
- 多团队协作开发不同模块的场景
- 需要运行时热插拔的系统

不适用于：
- 嵌入式或极端低延迟场景
- 仅有 2-3 个模块且不会增长的系统

### 5.3 经验教训

从透闪石的迁移实践中，总结以下经验：

1. **先改造数据交互，再改造控制流**：锁重入问题的修复（`do_compress()` → `compress_entries()`）可以在不改变模块名称的情况下独立完成，降低迁移风险。
2. **双模式优于大爆炸重写**：旧模块不需要为新协议修改任何代码。双模式意味着可以有一个可运行的过渡态，而不是在代码分支上等开发完成。
3. **作者声明不是装饰**：在调试阶段，作者信息帮助快速定位「这是谁写的模块」；在部署阶段，作者签名可扩展为代码签名验证的基础。

---

## 6. 结论

能力路由此前多应用于微服务和服务网格领域。本文将这一模式引入智能体模块化架构，提出了一套以服务声明为核心、标准消息信封为载体的模块间通信协议，并在透闪石系统中完成了从单体硬编码到能力路由的渐进迁移。

实验表明，该模式消除了类型侵蚀、锁重入和注入耦合三类典型问题，同时保持了与旧模块的二进制兼容。协议的开销（~0.1-0.5ms/次）在 LLM 应用场景中可忽略。
